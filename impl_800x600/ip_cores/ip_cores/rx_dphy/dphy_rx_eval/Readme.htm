QUEUE 		User-Input Selects the buffer control implementation.

  • PINGPONG - uses 2 FIFOs alternately to prevent the stored data from being overwritten by the next incoming high speed data transaction.

  • SINGLE - uses 1 circular FIFO. Read from FIFO continues until the empty flag asserts. This is for applications where there are large enough interval between high speed data arrival thus the incoming data will not be appended to the previous one.

  • QUEUE - uses 1 circular FIFO and a separate list of entries (queue) to track the number of cycles of each high speed transactions. The FIFO depth must be large enough to hold multiple D-PHY packets, including the hs-zero and trail bits.

    • RX FIFO Packet Delay 0-4096 User-Input Delays reading from the FIFO. The delay counter starts from the deassertion of the FIFO empty signal.
    The delay applies to every high speed data transaction.
    Increase the delay to compensate for any clock drift between the clock strobe and the freerunning byteclock domain.
    Ensure the arrival of the next high speed data will not overwrite the current one stored in the FIFO by adjusting the depth and the delay.
    RX FIFO Counter Width 0-12 User-Input Sets the width of the delay counter. This determines the maximum delay value for the design.
    RX FIFO Number of Queue Entries 2-6 User-Input Sets the depth of the queue entry list.
    Configure this to the maximum number of transactions that may be present in the data buffers at any given time.
    For example, in the case of the CSI-2 sequence, the line end, frame end and frame start may arrive while the last video line is still being read out of the data buffer. Therefore, the number of queue entries should be set to 4.

    Implementing the IP Module Using Lattice Diamond


    Instantiating the IP

    The core modules of the CSI-2/DSI DPHY Receiver IP are synthesized and provided in NGO format with black box Verilog source files for synthesis. A Verilog source file named <instance_name>_dphy_rx.v instantiates the black box of core modules. The top-level file <instance_name>.v instantiates <instance_name>_dphy_rx.v.

    The user does not need to instantiate the IP instances one by one manually. The top-level file along with the other Verilog source files are provided in <project_dir>. These files are refreshed each time the IP is regenerated.

    A Verilog instance template <instance_name>_inst.v or VHDL instance template <instance_name>_inst.vhd is also provided as a guide how to instantiate the generated soft IP module in their own top-level design.


    Hardware Evaluation

    An IP-specific license is required to enable full, unrestricted use of the CSI-2/DSI DPHY Receiver IP in a complete, top-level design. The CSI-2/DSI DPHY Receiver IP is available free or charge. Please request your free IP license at:
    http://www.latticesemi.com/licenseprocessing/ipcore_lic_req.cfm?api=true

    You may download or generate the CSI-2/DSI DPHY Receiver IP and fully evaluate through functional simulation and implementation (synthesis, map, place and route) without the IP license. The CSI-2/DSI DPHY Receiver IP also supports Lattice’s IP hardware evaluation capability, which makes it possible to create versions of the IP that operate in hardware for a limited time (approximately four hours) without requiring an IP license. However, the IP license is required to enable timing simulation, to open the design in Diamond EPIC tool, or to generate bitstreams that do not include the hardware evaluation timeout limitation.

    Running Functional Simulation

    To run simulation using Active HDL:

    1. Create new project using Lattice Diamond for Windows.
    2. Open Active-HDL Lattice Edition GUI tool.
    3. To customize the testbench parameters, edit the file tb_setup_params.v inside <project_dir>\<instance_name>\dphy_rx_eval\testbench folder.
      See the additional info related to Testbench Compiler Directives below.
    4. Click Tools -> Execute macro, then select <instance_name>_run.do file inside <project_dir>\<instance_name>\dphy_rx_eval\<instance_name>\sim\aldec .
    5. Wait for simulation to finish.

    Synthesizing and Implementing the IP


     In Clarity Designer, the Clarity Designer project file (.sbx) is added to Lattice Diamond as a source file after IP is generated. All required Verilog source files for implementation are invoked automatically. The IP can be directly synthesized, mapped and placed/routed in the Diamond design environment after the IP is generated. Note that default Diamond strategy (.sty) and default Diamond preference file (.lpf) will be used. When using the .sbx approach, import the recommended strategy from <project_dir>\dphy_rx_eval\<implementation_name>\impl\<device>\lse or <project_dir>\dphy_rx_eval\<implementation_name>\impl\<device>\synplify directories and set them as active strategy. A sample lpf file is also available in the same folder.

    Push-button implementation of this top-level design with either Lattice Synthesis Engine (LSE) or Synopsys Synplify Pro RTL synthesis is supported via the pre-built Diamond project file <implementation_name>_top.ldf located in <project_dir>\dphy_rx_eval\<implementation_name>\impl\<device>\lse or <project_dir>\dphy_rx_eval\<implementation_name>\impl\<device>\synplify directories.

     

    To use the pre-built project file:

    • Select Open -> Project under the File tab in Diamond.
    • Browse to <project_dir>\dphy_rx_eval\<implementation_name>\impl\<device>\<synplify|LSE>in the Open Project dialog box.
    • Select and open <implementation_name>_top.ldf . At this point, all of the files needed to support top-level synthesis and implementation will be imported to the project.
    • A sample lpf file is available in the \impl\<device>\<synplify|LSE> directory <implementation_name>_top.lpf.
    • Implement the complete design via the standard Diamond GUI flow, keeping all other options at their default values.
